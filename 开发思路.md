# 开发思路

## 总体思路

本项目按照"自底向上、逐层构建"的原则开发，分为三大模块、六个开发阶段。模块间存在上下游依赖关系：

```
模块2（网络建模） → 模块3（风险识别） → 模块4（AI智能体处置）
    ↑                                           |
    └───────────── 模型更新反馈 ←────────────────┘
```

**核心设计理念**：网络模型是底座，风险评估是中间件，AI智能体是上层应用。三者通过标准化数据接口串联，形成"网络建模→风险识别→智能处置→模型更新"的全链路闭环。

**验证策略**：以"汽车发动机供应链"为贯穿案例（详见附录A），每个阶段基于该案例的仿真数据验证功能实现，并输出科研图表用于汇报展示。

## 阶段一：基础设施与网络本体模型（模块2.1）

**目标**：搭建项目骨架，定义供应链网络核心数据结构，并用汽车发动机案例实例化验证。

**开发内容**：

1. **项目初始化**
   - 创建目录结构、安装依赖、配置文件
   - 定义全局配置（`config/settings.yaml`）
   - 创建图表样式模块（`src/visualization/style.py`）

2. **网络本体模型**（`src/network/models.py`）
   - 定义节点数据类：含节点类型（原材料供应商/零部件供应商/分系统集成商/总装厂）、属性（供货周期、产能上限、地理区位、可替代性等级）
   - 定义边数据类：含边类型（物料供应/信息传递/资金流转）、属性（供应量、依赖强度）
   - 定义 `SupplyChainNetwork` 核心类，封装 networkx 图操作

3. **多层耦合网络**（`src/network/multilayer.py`）
   - 实现物料依赖层、供应商协作层、物流运输层三层网络
   - 定义层间映射关系与耦合约束

4. **案例实例化**（`data/cases/auto_engine/`）
   - 将汽车发动机供应链案例（附录A）录入为 JSON/CSV 数据
   - 用案例数据实例化网络对象，验证模型完整性

**产出图表**：
- F1-1 供应链网络拓扑结构图：以分层布局展示案例网络全貌（25个节点、4层级）
- F1-2 多层耦合网络结构图：分别展示物料/协作/物流三层子网络

**验收标准**：
- 汽车发动机案例网络（25节点、4层级）能正确实例化、序列化/反序列化
- 输出的拓扑图中节点分层清晰、边方向正确、多层关系可区分

## 阶段二：参数化网络构造与拓扑分析（模块2.2 + 2.3）

**目标**：实现自动化网络生成与结构分析能力，用案例网络验证全部分析指标。

**开发内容**：

1. **参数化网络构造**（`src/network/builder.py`）
   - 实现网络构造算法，支持参数：网络规模、层级深度、供应商集中度、网络冗余度
   - 确保生成的网络满足供应链统计特征（幂律度分布、有向无环、局部聚类）
   - 支持批量生成不同配置的网络场景
   - 同时支持从案例数据文件直接加载真实网络

2. **拓扑特征分析**（`src/network/topology.py`）
   - 计算核心指标：度中心性、介数中心性、聚类系数、网络直径、平均路径长度
   - 输出结构化的拓扑分析报告（JSON + CSV 表格）

3. **脆弱性评估**（`src/network/vulnerability.py`）
   - 实现随机故障与定向攻击的抗毁性分析（节点/边移除模拟）
   - 计算 HHI 供应集中度指数
   - 替代供应路径可达性分析
   - 输出关键节点清单、薄弱环节排序与脆弱性评分报告

4. **网络可视化**（`src/visualization/network_vis.py`）
   - 多层网络拓扑可视化，关键节点高亮标注
   - 按中心性大小映射节点尺寸，按依赖强度映射边粗细

**产出图表**：
- F2-1 节点中心性分布图：对案例网络的度中心性、介数中心性排名柱状图，预期"动力总成系统集成商"和"ECU"介数中心性最高
- F2-2 网络度分布图：双对数坐标散点图，验证幂律分布特性
- F2-3 抗毁性衰退曲线：随机故障 vs 定向攻击对比折线图，预期定向攻击衰退更快
- F2-4 供应集中度热力图：按层级×物料类别展示 HHI 指数，预期"芯片晶圆"和"稀土"所在列 HHI 极高

**验收标准**：
- 案例网络拓扑指标计算结果符合预期（如芯片/稀土相关节点中心性高）
- 参数化构造器生成的 100 节点网络满足有向无环、幂律度分布约束
- 抗毁性分析产出完整衰退曲线，定向攻击衰退显著快于随机故障

## 阶段三：风险因素体系与量化指标（模块3.1）

**目标**：建立风险分类框架和可量化的评估指标，在案例上计算所有指标并排序。

**开发内容**：

1. **风险因素分类体系**（`src/risk/factors.py`）
   - 定义四类核心风险：材料短缺、供应商集中、物流中断、需求波动
   - 构建风险因素分类树（数据类层次结构）

2. **风险量化指标**（`src/risk/indicators.py`）
   - 为每类风险设计量化指标集
   - 基于网络拓扑数据计算各指标值
   - 实现 FMEA 风险模式与影响分析（发生概率 × 影响严重度 = RPN）

3. **仿真数据生成**（`src/simulation/data_generator.py`）
   - 为案例网络中的每个节点生成仿真运营数据（产能利用率、库存水位、交货准时率等）
   - 数据中嵌入已知风险特征（如芯片供应商产能利用率 > 95%、稀土供应商数量 = 1）
   - 生成 6 期（模拟 6 个季度）周期性数据，用于后续趋势分析

**产出图表**：
- F3-1 四类风险因素 RPN 排序图：水平柱状图，预期"芯片晶圆材料短缺"和"稀土供应商集中"RPN 最高
- F3-2 FMEA 风险矩阵图：散点气泡图（X轴=发生概率，Y轴=影响严重度，气泡大小=RPN），展示各风险因素在矩阵中的分布

**验收标准**：
- 四类风险均有对应的量化指标并成功计算
- RPN 排序结果与案例中预设的风险严重程度一致（芯片短缺 > 稀土集中 > 物流脆弱 > 需求波动）
- 仿真数据中嵌入的已知风险能被指标体系正确识别

## 阶段四：风险传播模型与综合评估（模块3.2 + 3.3）

**目标**：实现风险传播模拟和多方法融合的综合评估，用案例场景验证传播规律和评估准确率。

**开发内容**：

1. **SIR 风险传播模型**（`src/risk/propagation.py`）
   - 在供应链网络上实现改进 SIR 模型（正常→受影响→恢复）
   - 定义传播概率（依赖强度、替代路径）和恢复速率（应急能力、库存缓冲）
   - 实现三类扰动场景并映射到案例：
     - 单点中断：芯片晶圆供应商停供 → 观察 ECU → 电子电气系统 → 总装厂的传播链
     - 区域中断：华东区域供应商集体受灾 → 观察多条传播路径同时激活
     - 需求冲击：总装厂订单骤增 50% → 观察牛鞭效应沿供应链逐级放大
   - 输出影响范围、影响深度、影响持续时间

2. **模糊综合评价**（`src/risk/fuzzy_eval.py`）
   - 构建隶属度函数，将定性因素转为模糊评价矩阵
   - 实现多级模糊综合评价

3. **贝叶斯网络风险推断**（`src/risk/bayesian.py`）
   - 构建风险因素因果关系 DAG：材料短缺→产能不足→交付延迟
   - 用仿真数据训练条件概率表
   - 支持在部分信息缺失下的风险推理

4. **综合评估与融合**（`src/risk/assessment.py`）
   - 融合模糊评价与贝叶斯推断结果
   - 输出风险热力图、节点风险评分排序、风险等级（红/橙/黄/绿）

5. **仿真验证实验**（`src/simulation/scenarios.py`）
   - 构造已知风险场景（在案例网络中预埋明确的风险事件）
   - 运行评估流水线，对比识别结果与预埋标签
   - 计算识别准确率、漏报率、误报率

**产出图表**：
- F4-1 SIR 传播动态曲线：芯片停供场景下 S/I/R 三状态随时间步演化折线图
- F4-2 级联效应传播快照：案例网络在 t=0, t=5, t=10, t=20 的节点状态着色图
- F4-3 三类扰动场景对比图：分组柱状图，对比影响范围/深度/持续时间
- F4-4 风险热力图：案例网络节点着色（红/橙/黄/绿），综合评估结果
- F4-5 贝叶斯网络因果图：展示风险因素间的 DAG 结构与条件概率
- F4-6 模糊评价雷达图：案例中关键节点的多维度风险模糊评价
- F4-7 准确率验证报告表：混淆矩阵 + 准确率/漏报率/误报率统计

**验收标准**：
- 芯片停供场景：传播路径正确经过 ECU → 电子电气系统 → 总装厂
- 区域中断场景：多条传播路径同时激活，影响范围显著大于单点
- 需求冲击场景：观察到沿供应链逐级放大的牛鞭效应
- 风险识别准确率 ≥ 90%（基于预埋风险标签验证）
- 风险评估覆盖 4 类以上风险

## 阶段五：AI 智能体核心能力（模块4.1 + 4.2）

**目标**：实现基于 LLM 的瓶颈诊断和风险预警策略推荐，用案例的评估结果验证智能体输出质量。

**开发内容**：

1. **供应链知识库**（`src/agent/knowledge_base.py`）
   - 整理历史风险事件案例（如 2021 全球芯片短缺、2020 COVID 物流中断等真实事件）
   - 行业处置经验和供应链管理规则
   - 结构化知识存储（向量数据库 + 文本索引）
   - 知识增量更新接口

2. **RAG 瓶颈诊断**（`src/agent/rag_diagnosis.py`）
   - 接入通义千问 API（dashscope）
   - 实现 RAG 流程：风险报告 → 知识检索 → LLM 推理 → 结构化诊断报告
   - 诊断报告包含：瓶颈位置、成因分析、影响范围、严重程度
   - 诊断结果与传播模型交叉验证

3. **风险预警**（`src/agent/early_warning.py`）
   - 基于 6 期仿真数据的趋势分析
   - 多层级预警阈值（关注/警告/严重）
   - 验证：第 1-3 期数据逐步恶化 → 系统应在第 3-4 期触发预警

4. **策略推荐**（`src/agent/strategy.py`）
   - 构建防控策略知识图谱（策略 ↔ 风险类型 ↔ 网络特征）
   - 根据风险场景自动推荐策略组合
   - 策略可行性量化评估与优先级排序

**产出图表**：
- F5-1 多期风险趋势折线图：6 期仿真数据中关键节点风险评分变化 + 预警阈值线标注
- F5-2 策略推荐效果对比图：对案例中芯片短缺风险，对比"增加备选供应商"vs"增加安全库存"等策略的风险降低效果

**验收标准**：
- RAG 诊断能正确定位案例中的芯片供应瓶颈并关联到真实芯片短缺事件
- 预警系统在仿真数据连续恶化时正确触发分级预警
- 推荐策略与风险类型匹配，可行性评分合理

## 阶段六：多智能体协同与系统集成（模块4.3 + 集成）

**目标**：实现多智能体协同决策，完成全链路闭环集成，用案例进行端到端验证。

**开发内容**：

1. **领域智能体**（`src/agent/agents/`）
   - 库存风险感知智能体：安全库存优化、物料储备规划
   - 物流异常分析智能体：运输路线优化、物流资源调配
   - 需求预测智能体：需求趋势预测、订单波动预警
   - 协调智能体：跨领域方案整合与冲突消解

2. **多智能体协同**（`src/agent/multi_agent.py`）
   - 智能体间通信协议与交互接口
   - 冲突检测与多目标权衡机制
   - 综合处置方案生成

3. **反馈闭环**
   - 处置结果 → 案例库存储
   - 处置结果 → 网络模型更新（节点属性、风险状态）
   - 形成完整闭环：网络建模 → 风险识别 → 智能处置 → 模型更新

4. **端到端验证**
   - 输入：汽车发动机案例网络 + 芯片停供场景
   - 全流程：网络分析 → 风险评估 → 瓶颈诊断 → 预警触发 → 多智能体协同处置 → 网络更新
   - 验证处置后网络风险评分下降

**产出图表**：
- F6-1 多智能体协同架构图：展示 4 类智能体角色、职责与交互流程
- F6-2 处置方案多目标评估雷达图：成本/效率/风险降低/实施难度多维度评分
- F6-3 全链路闭环验证流程图：端到端数据流与反馈闭环
- F6-4 处置前后风险对比图：案例网络处置前后的风险热力图对比

**验收标准**：
- 实现 3 个领域智能体（库存/物流/需求）
- 多智能体能协同生成不冲突的综合处置方案（如库存智能体建议增加芯片安全库存，采购智能体建议引入第二供应商，两者不冲突）
- 端到端流水线可运行：案例网络 + 芯片停供 → 完整处置报告 → 网络更新后风险评分下降

## 各阶段依赖关系

```
阶段一 → 阶段二 → 阶段三 → 阶段四 → 阶段五 → 阶段六
 网络      拓扑      风险      传播      AI       多智能体
 本体      分析      因素      评估     诊断      协同
 +案例    +案例验证  +仿真数据  +场景验证  +趋势验证  +端到端
```

- 阶段一是所有后续阶段的基础（含案例数据）
- 阶段三依赖阶段二的网络拓扑数据，同时开始生成仿真数据
- 阶段四使用阶段三的仿真数据验证识别准确率
- 阶段五使用阶段三生成的 6 期数据验证预警功能
- 阶段六整合所有模块，用案例进行端到端闭环验证

## 附录A：汽车发动机供应链案例

### A.1 案例选取理由

选择汽车发动机供应链作为验证案例，原因如下：
- **层级清晰**：天然具有原材料→零部件→分系统→总装的 4 层级结构，与研究定义的节点类型完全匹配
- **规模适中**：约 25 个核心节点，便于可视化展示和实验分析
- **风险真实**：2021 全球芯片短缺、稀土供应集中、COVID 物流中断等真实事件可直接映射为仿真场景
- **文献充分**：汽车供应链是供应链管理研究的经典对象，便于与已有成果对标

### A.2 网络结构定义

```
第0层 - 总装厂（1个节点）
  └── OEM: 整车总装厂（长春）

第1层 - 分系统集成商（3个节点）
  ├── T1-P: 动力总成系统集成商（上海）
  ├── T1-C: 底盘系统集成商（武汉）
  └── T1-E: 电子电气系统集成商（深圳）

第2层 - 零部件供应商（9个节点）
  ├── T2-E1: 发动机缸体（长沙）        → T1-P
  ├── T2-E2: 涡轮增压器（无锡）        → T1-P
  ├── T2-T1: 变速箱总成（重庆）        → T1-P
  ├── T2-B1: 制动系统（烟台）          → T1-C
  ├── T2-S1: 悬挂系统（柳州）          → T1-C
  ├── T2-W1: 转向器（苏州）            → T1-C
  ├── T2-ECU: ECU控制单元（深圳）      → T1-E
  ├── T2-H1: 线束总成（天津）          → T1-E
  └── T2-SN: 传感器模组（合肥）        → T1-E

第3层 - 原材料供应商（12个节点）
```
